# 実装ルール（Image to Mesh Web）

このリポジトリで開発（人間 / AI アシスタント問わず）する際のルール。

## 役割分担

- **git のコミット・プッシュはユーザーが行う。** アシスタントはコード変更・コメント追加・
  ドキュメント追記まで。指示なくコミットしないこと。
- **ブラウザでの実機確認はユーザーが行う。** ルートの `run.bat` がローカルサーバ
  （`python -m http.server`、既定ポート 8000、空きポートへ自動退避）を起動して
  ブラウザを開く。アシスタントに実機確認は求めない。

## 技術条件

- **GitHub Pages でホストできる静的構成のみ。** ビルド工程・サーバサイド処理を持ち込まない。
- **商用利用可能なライセンスのみ。** 現状は自前実装 + MIT。外部ライブラリ・外部モデル
  （weights 含む）を導入する場合は、導入前にライセンスを確認して README に記載する。

## 進め方

- **トークンを節約し、段階的に開発する。** 一度に大きく変えず、1ステップずつ検証してから次へ。
- **md は増やしすぎない。** root の md は原則 `CLAUDE.md` / `README.md` / `PIPELINE.md` /
  `PROGRESS.md` の4本に限定する。新しい計画専用 md は、ユーザーが明示した場合だけ作る。
- **仕様と履歴を分ける。** 現行仕様は `README.md` と `PIPELINE.md` に集約し、失敗案・検討ログ・
  旧仕様は `docs/archive/` に移す。完了済みの計画 md は root に残さない。
- **`PROGRESS.md` は現在状況だけに保つ。** 長い実装ログや試行錯誤を追記し続けず、現行構成、
  実装済み、確認待ち、直近変更の短い要約に留める。
- **js を変更したら `index.html` のキャッシュバスター（`?v=YYYYMMDD-N`）を更新する。**

## エンコーディング

- **文字化けした状態で読めたつもりになって進めない。** `CLAUDE.md` やソース、md を読む際に
  日本語コメント・本文が文字化けして見える場合は、まず UTF-8 を明示するなど読み取り方法を変えて
  正常に読めるまで試すこと。
- **正常に読めない場合は処理を止めて報告する。** エンコーディングを変えても内容を確認できない場合、
  推測で編集・実装を続けず、どのファイルが読めないかと試した方法をユーザーに報告して指示を待つこと。
