# 実装ルール（Image to Mesh Web）

このリポジトリで開発（人間 / AI アシスタント問わず）する際のルール。

## 役割分担

- **git のコミット・プッシュはユーザーが行う。** アシスタントはコード変更・コメント追加・
  ドキュメント追記まで。指示なくコミットしないこと。
- **ブラウザでの実機確認はユーザーが行う。** ルートの `run.bat` がローカルサーバ
  （`python -m http.server`、既定ポート 8000、空きポートへ自動退避）を起動して
  ブラウザを開く。アシスタントに実機確認は求めない。

## 技術条件

- **GitHub Pages でホストできる静的構成のみ。** ビルド工程・サーバサイド処理を持ち込まない。
- **商用利用可能なライセンスのみ。** 現状は自前実装 + MIT。外部ライブラリ・外部モデル
  （weights 含む）を導入する場合は、導入前にライセンスを確認して README に記載する。

## 進め方

- **トークンを節約し、段階的に開発する。** 一度に大きく変えず、1ステップずつ検証してから次へ。
- **計画と経緯を md に残す。** 設計判断・実装ログ・失敗の教訓は `PLAN_INPAINT.md` などの
  計画ドキュメントへ日付付きで追記する。実装状況の一覧は `PROGRESS.md`、
  ユーザー向け仕様は `README.md` を更新する。
- **ロジックは Node の合成テストで検証してから結線する。** ブラウザ非依存の処理
  （深度・マスク・メッシュ演算など）は、実機に頼らず合成データのテストを scratchpad に
  書いて PASS を確認する。実画像での見た目の確認はユーザーのブラウザテストに委ねる。
- **js を変更したら `index.html` のキャッシュバスター（`?v=YYYYMMDD-N`）を更新する。**
