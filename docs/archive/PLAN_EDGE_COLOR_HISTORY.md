# エッジ色の改善計画（PLAN_EDGE_COLOR）

2026-07-05。■01（EdgeSnap + シーム分割）の実機フィードバックへの対応計画。
**コードは未変更。実装前の検討ドキュメント。**

## 1. 現象と原因

### 現象A: UVスナップによる色の伸び（最優先）
- 例1: 看板のフチがモデル解像度のブロック状に乱れる
- 例2: 空（Sky Backdrop）と屋根の境界で縦スジ状のスミア

**原因**: EdgeSnap はランプ画素（最大 Snap Width px）の UV を吸着元台地画素の
**中心1点**へ差し替える（viewer.js の UV 差し替え）。
- 同じ吸着元を持つランプ帯全体が**同一 UV = 定数色**で塗られる → スジ/面積のある伸び
- 深度解像度（例 ~700×500）とテクスチャ解像度（元画像、例 2000×1500）が違うため、
  UV の移動が「モデル画素単位」の粗い格子で起き、テクスチャ上では数〜十数px の
  ブロックとして見える（看板の例）
- 隣接ランプ画素が別の吸着元を持つと、隣り合う定数色の帯になる（空の縦スジの例)

### 現象B: 奥インペイント（backfill）に手前の色が混入
**原因**: backfill.js `resampleColor` が元画像を**バイリニアでモデル解像度へ縮小**して
使うため、エッジ画素の色は手前/奥の混色になっている。プルプッシュの種
（far側エッジ + 襟4px）はまさにそのエッジ上にあり、混色を起点に色を引き延ばしている。
EdgeSnap は深度を分離したが、**backfill が使う色はスナップ前の混色のまま**。

## 2. 解決案の比較

### A案（推奨）: UVスナップ廃止 + テクスチャ空間の色パッチ（元画像解像度）
ジオメトリの UV は元のまま戻し、混色そのものを**元画像解像度のテクスチャ側で**
台地の色に置き換える。

1. **EdgeSnap の UV 差し替えを廃止**（深度スナップと srcRoot の記録は維持）
2. **混色帯マスクを画像解像度で生成**:
   - モデル解像度の「ランプ画素（srcRoot>=0）+ シーム両側1px」を最近傍アップスケール
   - 縮小バイリニアのにじみ半径分（`ceil(imgW/W)+1` px 程度）を膨張
3. **side 分類の引き継ぎ**: 各帯画素は srcRoot がどちらの台地（near/far）かを引き継ぐ
4. **side 別の色伝播**: 帯内を「帯外の自側画素のみ」を種に伝播して埋める
   （反復拡散 or 小規模プルプッシュ。帯幅は狭いので数パスで収束）。
   near 帯には near の色だけ、far 帯には far の色だけが入る
5. **パッチ済み画像を使う**: viewer の表示テクスチャ / GLB テクスチャ / backfill の
   入力 color をパッチ済みに差し替え → **現象A と B が同時に解決**

- 長所: UV は連続のまま（ブロック/スジが原理的に出ない）。テクスチャ解像度で
  処理するので粒度の問題がない。backfill の混色も根元から消える
- 短所: テクスチャを書き換える（「画像を変えない」原則から外れる。
  ただし Original ダウンロードは原本ファイルのまま。表示/エクスポート用のみパッチ）
- 演算量: 帯画素のみ処理（画像全体の数%。純JSで問題ない想定）

### B案: グラデーションUV（テクスチャ不変のまま緩和）
UV を定数コピーではなく**台地エッジで反射**させる: `uv' = 2·uv_root − uv`
（または「台地内側の数px の帯を引き伸ばして帯に割り当てる」圧縮リマップ）。
ランプ帯が台地**内部**の色を連続サンプルするため、定数色のスジは消える。

- 長所: viewer の UV 差し替え部の小変更のみ。テクスチャ不変
- 短所: 粒度がモデル画素のままなので、看板のようなクッキリした模様では
  まだ乱れ（鏡像の写り込み）が見える。**現象B は解決しない**（別途下記の対応が必要）

### C案: 最小変更（UVスナップの適用幅を1pxに限定）
シーム直近の1画素だけ UV を差し替え、それより外側のランプは元 UV のまま。

- 長所: 数行の変更。伸びの面積が最小になる
- 短所: 混色の除去も1px分だけになり、エッジのにじみが残る。暫定策

### backfill 単体の混色対策（B/C案を選ぶ場合に必要。A案なら不要）
- 種の色を「エッジ上の混色画素」ではなく、
  (a) EdgeSnap の srcRoot（吸着元台地画素）の色、または
  (b) 種から bgDisp 勾配に沿って 2〜3px far 側へ潜った画素の色にする
- いずれも backfill.js の色初期化（`pullPush` の種色）だけの変更で済む

## 3. 推奨実装順序（A案）

1. viewer.js: uvSrcIndex による UV 差し替えを外す（uvSrcIndex 自体は色パッチが使う）
2. `js/colorpatch.js` 新規: 入力 = 元画像 ImageData + srcRoot + シーム情報（モデル解像度）、
   出力 = パッチ済み ImageData。純データ入出力で Node 合成テストを先に書く
   （合成: 2色の台地 + 混色帯 → 帯が各側の色だけで埋まる / 帯外は不変 / side 越境ゼロ）
3. main.js: パッチ済み画像を Viewer.setData の colorTex と backfill の color に配線
4. 実機確認（ユーザー）: 看板のフチ / 空の境界 / 覗き込み時の backfill 色
5. 不足が残る場合のみ B案（反射UV）の併用を検討

## 3.4 A案の実装（2026-07-05 実装済み・実機検証待ち）

暫定実験（3.5）の結果「あまりよくない」（混色フリンジが目立つ）との判断で A案を実装。

- `js/colorpatch.js` 新規: 上記 2.A のとおり。実装上のポイント:
  - side ラベルは明示的な near/far 分類ではなく**スナップ後深度そのもの**を使い、
    「相対深度差 <= 0.10 の隣接画素からしか色を受け取らない」制約で伝播
    → シームを色が越えず、任意個の台地/シームに一般化（成分分割不要）
  - 帯 = ランプ画素（srcRoot>=0）+ 深度段差対（>0.10、シーム分割と同じ・非連動）の両側
  - 伝播は「同じ側の確定隣接画素の平均色」をパス一括適用で内側へ（EdgeSnap と同じ構造）
  - 未解決の孤立帯画素は元の色のまま。元画像 ImageData は非破壊（コピーに書く）
- `js/main.js`: recompute で ColorPatch を適用（Edge Threshold=Off 時はスキップ）。
  パッチ済み画像を表示テクスチャと **backfill の入力 color の両方**に使用
  → 現象A（伸び）と現象B（インペイントへの手前色混入）を同時に解消
- UV は 3.5 の状態（完全に元のまま）を維持。ジオメトリ・深度は不変
- Node 合成テスト（scratchpad/test_colorpatch.js）: 混色帯が各側の色だけで埋まる /
  帯外は不変 / alpha保持 / 入力非破壊 / 平坦→null / srcRoot なし（段差対のみ）でも動作。全 PASS

## 3.5 暫定実験（2026-07-05 実装済み → A案に置き換え）

A案着手前に、ユーザー指示で「**深度とカラーの UV を分離し、カラーは完全に元のまま**」を
実機確認する（手前と奥の色がエッジで混ざるが、違和感の程度を見る）:

- viewer.js の UV 差し替え（吸着元へのコピー）を**廃止** → 全頂点が元画像位置の UV
- シーム分割の複製頂点の UV も「プレート側からコピー」をやめ、**複製元の角の UV のまま**
  → 延長セルも元画像の該当位置の色（混色を含む）をサンプルする
- EdgeSnap の uvSrcIndex は未使用になるが、将来の色パッチ（A案）用に配線は残す
- 期待される見え方: ブロック/スジは消える。代わりに両プレートのフチに
  元画像由来の混色フリンジ（にじみ）が乗る
- 問題が大きければ git で本節以前の状態に戻す

## 3.6 実機フィードバック1回目とフリンジ修正（2026-07-05）

A案適用後の実機確認: backfill のボケは背景系の色のみになり（手前の赤の混入は解消）、
残った問題は**穴のフチの赤 / ポール横の白のフリンジ**。

- **原因**: シーム分割の複製頂点の UV を暫定実験（3.5）で「複製元の角のまま」に
  していたため、プレートの1セル延長面が**相手側**のテクスチャ位置をサンプルしていた
  （far プレートの延長が手前の看板の赤を拾う等）。
- **修正**: 複製頂点の UV を「延長元（同じプレート側）の角からコピー」へ戻した
  （viewer.js addDuplicate）。テクスチャは ColorPatch 済みなので延長面は自分側の純色になる。
- ランプ頂点の UV は元のまま（3.4 の構成）。変更は複製頂点の UV のみ。

残件: backfill 色のプルプッシュ由来のスミア（グレーのボケ）は既知の限界で、
■02（インペイント化）の対象。

## 3.7 未解決画素フォールバックの実験とロールバック（2026-07-05）

実機フィードバック2回目: 稜線の先端などに黒点が残る。コンソールの
`[ColorPatch] unresolved: 292` から「同じ側に届かない帯画素が混色のまま残る」ことが
原因と推定し、「最も深度が近い確定隣接画素で埋める」フェーズ2を実装した。

**結果: フェーズ2は 292 画素を埋めた（fallbackPx:292, unresolved:0）が、
見た目の黒点は変わらず → 黒点の原因は ColorPatch の未解決画素ではない。ロールバック済み。**

改めての原因候補（未検証）:
1. **EdgeSnap の unresolved（このシーンで 111 画素）**: ランプ幅が Snap Width=4 を超えた
   画素は中間深度のまま残る仕様。稜線のような広いボケエッジでは、その画素が
   シーム分割で「小さな中間深度の板片」になり、空中に浮く黒点として見える可能性が高い。
   → **検証方法: Snap Width を 12〜16 に上げて黒点が減るか見る**（コード変更不要）。
   Points Only 表示で黒点が点として浮いているかも判別材料になる。
2. backfill レイヤーの縁（種・襟）の色や押し込みの露出。

つまり黒点は「色の問題」ではなく「幾何（中間深度の残骸）の問題」の可能性が高い。
対処するなら ColorPatch ではなく EdgeSnap 側（unresolved 画素の扱い）になる。
